Mezzanine in a nutshell

  • A toolkit, built on top of Django, designed for building sites with editable content
  • A toolkit with a somewhat opinionated base template ... that you are free to discard if you don't like it
  • BSD Licence
  • Supports Django 1.7, 1.8
  • Supports Python 2.6, 2.7, 3.3+
  • Stephen McDonald
  • mezzanine.jupo.org
  • toolkit, distinct from CMS. Set of APIs that you put together rather than a full suite you can configure
  • opinionated in the same way that the Django default template is. Useful to start with
  • Versions are for latest stable, 4.0.1 at time of writing. 3.1.x series supports Django 1.6
  • 2-clause BSD

Getting started


apt-get install python3 python3-dev virtualenvwrapper \
	libjpeg8 libjpeg8-dev
mkvirtualenv --python=`which python3` my_first_mezzanine_site
pip install mezzanine
mezzanine-project my_first_mezzanine_site

Seem familiar?

  • This assumes python3 on Ubuntu
  • python3-dev, libjpeg required for building Pillow
  • A few other libraries required if you want Pillow to support formats other than PNG + JPEG, see the Pillow docs
  • Almost identical to getting started with Django

Mezzanine Features

  • Preview unpublished content
  • In-page editing
  • Multi-lingual sites

Mezzanine Features

Built on top of Django

  • Uses existing Django components where it makes sense
    • Models, forms..
    • Migrations, the ORM...
  • Provides extended version of Django components where it makes things easier
  • Use the existing Django tooling for deploy etc

Mezzanine Features

Structure is separate from content

  • Structure is your code
  • Edit it with your favorite tools: emacs/Vim/Atom... magnetized needle/butterfly
  • Keep it in your version control system
  • Test it with your CI system
This may not seem particularly groundbreaking, but if you compare it to Drupal/Wordpress where site structure is in the database with content...

Two techniques for managing content

Pages vs Objects

Page-based

  • Small (~10) number of objects
  • Manual control over position in site hierarchy
  • Appear in menus
  • Objects can appear in menus, just a bit more work involved

Page-based

Implementation

  • Subclass mezzanine.pages.models.Page
  • Don't need to write your own view
  • Or just use mezzanine.pages.models.RichTextPage

Page-based

Implementation - models.py


from mezzanine.pages.models import Page, RichText

class Employee(Page, RichText):

    position = models.CharField(max_length=60)
    photo = FileField(blank=True)

Page-based

Implementation - admin.py


from mezzanine.pages.admin import PageAdmin

admin.site.register(models.Employee, PageAdmin)

Page-based

Aside: Template look-up order

  • pages/[Page slug].html
  • pages/[Class name].html
  • pages/[Parent slug]/[Page slug].html
  • pages/[Parent slug]/[Page slug]/[Class name].html
  • pages/[Parent slug]/[Class name].html
  • pages/page.html

Page-based

Aside: Template look-up order

eg, directors/big-boss, type Employee

  • pages/big-boss.html
  • pages/employee.html
  • pages/directors/big-boss.html
  • pages/directors/big-boss/employee.html
  • pages/directors/employee.html
  • pages/page.html

Object-based

  • When you expect to have a lot of it
  • Products, publications...
  • Where you expect people to find them by search or a list page
  • Not in menus
  • Subclass mezzanine.core.models.Displayable
  • Need to define your own view logic
  • More than you want to manage by hand
  • Not in menus unless you work for it
  • Page is a subclass of Displayable

Object-based

Implementation - models.py


class Widget(Displayable, RichText):

    category = models.ForeignKey(WidgetCategory)

    def get_absolute_url(self):
        return self.category.get_absolute_url() + self.slug

Object-based

Implementation - admin.py


from mezzanine.core.admin import DisplayableAdmin
import copy

widget_fieldsets = copy.deepcopy(DisplayableAdmin.fieldsets)
widget_fieldsets[0][1]["fields"].insert(1, "category")
widget_fieldsets[0][1]["fields"].insert(1, "content")


class WidgetAdmin(DisplayableAdmin):

    fieldsets = widget_fieldsets
    inlines = (TechnicalDocumentInline,)

admin.site.register(models.Widget, WidgetAdmin)

Things Mezzanine does well

Config is not Content

  • Structure is not content
  • Config, Structure should be versioned, replicable
  • Content is dynamic
  • Define your content and structure in code, store it in version control
  • Everything else is content, store it in a database
  • Continuous Integration!
  • Repeatable Deploys!

Things Mezzanine does well

Control your markup

Things Mezzanine does well

Control your markup

  • Uses the Django template system
  • Doesn't enforce any particular structure on the markup
  • Default templates suggest a sensible approach, override them if you want

Things Mezzanine does well

Timed publish and expire

  • Built in
  • Just works

Things Mezzanine does well

Drop in addition to existing site

  • Nice web app, how do you update it?
  • Mezzanine is just another Django app, retrofitting is pretty straight forward
  • Template tweaking is usually the hardest part

Comparison

Things Mezzanine doesn't do so well

Things Mezzanine doesn't do so well

Versioned content

  • No support in core Mezzanine
  • No generic solution (that I know of)
  • Could roll your own, not easy
  • Something other CMS solutions support

Things Mezzanine doesn't do so well

Documentation

  • Curse of the moderately successful open-source project
  • Efforts underway to improve this
  • You can help!
  • Source is very readable...

Code changes are common

  • Not for organizations that find code deploys hard/scary
  • Structure is code, code changes often

Summary

  • Mezzanine: It's pretty much just Django, with some extra bits to make life easier
  • Quick to get started
  • Structure != Content
  • Strong ecosystem, many Django modules can be used
  • Pages vs Objects
  • Add it to your existing Django app
